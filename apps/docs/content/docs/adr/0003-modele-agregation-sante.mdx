# ADR 0003 : modèle commun d'agrégation de santé Docker/Kubernetes

## Statut

Proposé

## Contexte

Docker et Kubernetes exposent des informations de santé sous des formats différents (statut de conteneur vs statut de pod, structure d'API distincte). Le frontend a besoin d'un format unique pour afficher le statut de n'importe quelle ressource sans logique conditionnelle par orchestrateur.

## Décision

Définir un schéma pivot interne, par exemple :

```json
{
  "resource_id": "uuid",
  "status": "healthy | degraded | down | unknown",
  "orchestrator": "docker | kubernetes",
  "last_checked": "timestamp",
  "details": { "cpu": "...", "memory": "...", "restarts": 0 }
}
```

Chaque connecteur (Docker, Kubernetes) traduit sa réponse native vers ce schéma avant de la persister/pousser au frontend.

## Justification

- Découple le frontend de la source réelle de l'infrastructure
- Permet d'ajouter un futur orchestrateur (ex. Nomad) sans changer le frontend
- Simplifie la logique WebSocket : un seul format à sérialiser/pousser

## Conséquences

- Nécessite un mapping explicite par orchestrateur, à maintenir si leurs API évoluent
- Certains détails spécifiques à un orchestrateur peuvent être perdus dans la traduction (compensé par le champ `details` libre)
